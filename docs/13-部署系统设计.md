## 13.2 环境模型

### 环境层级

```
Project
  │
  ├── Development Environment (自动创建, 不可删除)
  │   ├── URL: <project>-dev.nova-studio.app
  │   ├── Branch: develop
  │   ├── Auto-deploy: 每次 commit
  │   └── 用途: 日常开发调试
  │
  ├── Preview Environment (可创建多个, 每个 PR 一个)
  │   ├── URL: <project>-pr-<number>.nova-studio.app
  │   ├── Branch: feature/*
  │   ├── Auto-deploy: PR 创建/更新时
  │   └── 用途: 代码审查、功能预览
  │
  ├── Staging Environment (可选, 0 或 1 个)
  │   ├── URL: <project>-staging.nova-studio.app
  │   ├── Branch: release/*
  │   ├── Auto-deploy: 手动触发
  │   └── 用途: 上线前验收测试
  │
  └── Production Environment (可选, 0 或 1 个)
      ├── URL: <project>.nova-studio.app 或 自定义域名
      ├── Branch: main
      ├── Auto-deploy: 手动触发 (防止误操作)
      └── 用途: 线上正式环境
```

### 环境配置

| 配置项 | Development | Preview | Staging | Production |
|---|---|---|---|---|
| 资源限制 | 256MB / 0.5 CPU | 256MB / 0.5 CPU | 512MB / 1 CPU | 1GB+ / 2+ CPU |
| 自动休眠 | 30min 无访问 | 7天无访问 | 否 | 否 |
| 环境变量 | dev_* | pr_* | staging_* | prod_* (加密) |
| 数据库 | 共享 Dev DB | 独立 Preview DB | 独立 Staging DB | 独立 Prod DB |
| 构建缓存 | 启用 | 启用 | 启用 | 启用 |
| CDN | 否 | 否 | 是 | 是 + 自定义域名 |
| 访问控制 | 公开 | 限团队成员 | 限团队成员 | 公开 (或 IP 白名单) |

## 13.4 多平台部署 Provider

### Provider 抽象接口

```
DeployProvider (接口)
  │
  ├── deploy(artifact, config): Promise<DeployResult>
  ├── rollback(deploymentId): Promise<RollbackResult>
  ├── getDeploymentStatus(deploymentId): Promise<Status>
  ├── deleteDeployment(deploymentId): Promise<void>
  ├── configureDomain(deploymentId, domain): Promise<DomainResult>
  └── getLogs(deploymentId): Promise<LogStream>
```

### 各 Provider 实现

#### 4.1 Vercel Provider

```
VercelProvider
  │
  ├── 认证方式: Vercel Access Token (用户绑定或平台共享)
  │
  ├── 部署流程:
  │   1. 调用 POST /v13/deployments
  │      - files[]: 构建产物文件列表
  │      - projectSettings: { framework: "nextjs", buildCommand, outputDir }
  │      - target: "production" | "preview"
  │      - env: 环境变量
  │   2. 轮询 GET /v13/deployments/{id} 直到 READY
  │   3. 获取 deployment URL
  │
  ├── 域名绑定:
  │   - 自动子域名: <project>-<env>.vercel.app
  │   - 自定义域名: POST /v9/projects/{id}/domains
  │   - Vercel 自动管理 SSL (Let's Encrypt)
  │
  ├── 回滚:
  │   - POST /v13/deployments/{id}/promote (将旧部署提升为 production)
  │
  └── 优势: 零配置 / Edge Network / 自动 SSL / Preview Deployments
```

#### 4.2 Cloudflare Pages Provider

```
CloudflarePagesProvider
  │
  ├── 认证方式: Cloudflare API Token + Account ID
  │
  ├── 部署流程:
  │   1. 上传构建产物到 Cloudflare R2 (临时存储)
  │   2. 调用 POST /accounts/{id}/pages/projects/{name}/deployments
  │      - directory: 构建产物路径
  │   3. 轮询部署状态直到 successful
  │
  ├── 域名绑定:
  │   - 自动子域名: <project>.pages.dev
  │   - 自定义域名: POST /accounts/{id}/pages/projects/{name}/domains
  │   - Cloudflare 自动管理 SSL (如果域名在 CF DNS)
  │   - 外部域名: 需用户手动配置 CNAME + 等待 SSL 签发
  │
  ├── 回滚:
  │   - POST /accounts/{id}/pages/projects/{name}/deployments/{id}/rollback
  │
  └── 优势: 全球 CDN / 无限带宽 (Pro) / Workers 集成 / R2 存储
```

#### 4.3 Docker Provider

```
DockerProvider
  │
  ├── 认证方式: Registry Credentials (Harbor / Docker Hub / ECR)
  │
  ├── 部署流程:
  │   1. 在 Sandbox 中构建 Docker 镜像
  │      - 生成 Dockerfile (基于项目框架模板)
  │      - docker build -t registry/builder-<project>:<sha> .
  │   2. 推送镜像到 Registry
  │      - docker push registry/builder-<project>:<sha>
  │   3. SSH 到目标服务器执行
  │      - docker pull <image>
  │      - docker stop <container> && docker rm <container>
  │      - docker run -d --name <project>-<env> \
  │          -p <port>:3000 \
  │          --env-file /etc/nova/<project>-<env>.env \
  │          --restart unless-stopped \
  │          <image>
  │   4. 配置 Nginx/Caddy 反向代理 + SSL
  │
  ├── 域名绑定:
  │   - 通过 Caddy 自动获取 Let's Encrypt 证书
  │   - 或通过 Nginx + Certbot
  │   - 反向代理: 域名 → 容器端口
  │
  ├── 回滚:
  │   - docker stop <container> && docker run <previous-image>
  │   - 保留最近 5 个镜像版本用于回滚
  │
  └── 优势: 完全控制 / 任意后端 / 自托管 / 私有化部署
```

#### 4.4 Kubernetes Provider

```
KubernetesProvider
  │
  ├── 认证方式: Kubeconfig / Service Account Token
  │
  ├── 部署流程:
  │   1. 构建镜像并推送到 Registry (同 Docker Provider)
  │   2. 生成 Kubernetes Manifests (Helm Chart 模板)
  │      ┌─ Deployment
  │      │   - replicas: 1 (dev) / 2+ (prod)
  │      │   - resources: limits + requests
  │      │   - envFrom: Secret + ConfigMap
  │      │   - livenessProbe: GET /api/health
  │      │   - readinessProbe: GET /api/ready
  │      │   - imagePullPolicy: Always
  │      ├─ Service
  │      │   - type: ClusterIP
  │      │   - port: 3000
  │      ├─ Ingress
  │      │   - host: <project>-<env>.nova-studio.app
  │      │   - tls: cert-manager 自动签发
  │      │   - annotations: cloudflare / nginx
  │      ├─ HorizontalPodAutoscaler (prod only)
  │      │   - minReplicas: 2
  │      │   - maxReplicas: 10
  │      │   - targetCPU: 70%
  │      ├─ Secret (环境变量, 加密)
  │      └─ ConfigMap (非敏感配置)
  │   3. helm upgrade --install <release> ./chart \
  │        --set image.tag=<sha> \
  │        --set env=<env> \
  │        --namespace <project>-<env>
  │   4. 等待 Rollout 完成
  │      - kubectl rollout status deployment/<release>
  │
  ├── 域名 & SSL:
  │   - Ingress Controller (Nginx / Traefik)
  │   - cert-manager + Let's Encrypt (HTTP-01 / DNS-01)
  │   - 自动续期 (cert-manager 自动处理)
  │   - Wildcard 证书: *.nova-studio.app
  │
  ├── 回滚:
  │   - kubectl rollout undo deployment/<release>
  │   - 或 helm rollback <release> <revision>
  │   - 滚动更新: 零停机回滚
  │
  ├── 滚动更新策略:
  │   - strategy: RollingUpdate
  │   - maxSurge: 1
  │   - maxUnavailable: 0
  │   - 新 Pod Ready 后才移除旧 Pod
  │
  └── 优势: 自动伸缩 / 自愈 / 零停机 / 生产级 / 多副本
```

### Provider 对比矩阵

| 特性 | Vercel | Cloudflare Pages | Docker | Kubernetes |
|---|---|---|---|---|
| 部署速度 | 极快 (10-30s) | 快 (15-40s) | 中 (30-60s) | 中 (30-90s) |
| 自动 SSL | ✓ 内置 | ✓ 内置 | 需 Caddy/Certbot | ✓ cert-manager |
| 自定义域名 | ✓ 免费 | ✓ 免费 | 需手动配置 | ✓ Ingress |
| 自动伸缩 | ✓ 内置 | ✓ Workers | 需手动 | ✓ HPA |
| SSR 支持 | ✓ 原生 | ✓ (Workers) | ✓ | ✓ |
| 私有化 | ✗ | ✗ | ✓ | ✓ |
| 成本 | $20+/月 | $20+/月 | 服务器成本 | 集群成本 |
| 回滚 | ✓ 一键 | ✓ 一键 | 镜像切换 | rollout undo |
| 日志流 | ✓ 实时 | ✓ 实时 | docker logs | kubectl logs |

## 13.5 域名管理

### 域名体系

```
三级域名体系:
  │
  ├── 1. 系统子域名 (自动分配, 免费)
  │   └── <project>-<env>.nova-studio.app
  │       例: nova-landing-prod.nova-studio.app
  │           nova-landing-dev.nova-studio.app
  │           nova-landing-pr-42.nova-studio.app
  │
  ├── 2. 自定义子域名 (用户绑定, 免费)
  │   └── <name>.nova-studio.app
  │       例: nova.nova-studio.app
  │       流程: 用户申请 → 系统验证可用 → 绑定到环境
  │
  └── 3. 自定义域名 (用户自有域名, 需 DNS 配置)
      └── www.example.com
          流程:
            1. 用户添加域名: POST /domains { domain, environmentId }
            2. 系统返回 CNAME 目标: <hash>.target.nova-studio.app
            3. 用户在 DNS 提供商配置 CNAME 记录
            4. 系统轮询 DNS 验证 (最多 10 分钟)
            5. 验证通过 → 自动签发 SSL 证书
            6. 域名状态: pending → verifying → active
```

### 域名状态机

```
┌─────────┐     ┌───────────┐     ┌──────────┐     ┌────────┐     ┌────────┐
│ pending │────▶│ verifying │────▶│ ssl_     │────▶│ active │     │ failed │
│         │     │ (DNS检查) │     │ issuing  │     │   ✓    │     │        │
└─────────┘     └─────┬─────┘     └──────────┘     └────────┘     └────────┘
                      │                                               ▲
                      │ DNS 验证失败                                    │
                      └───────────────────────────────────────────────┘
```

### SSL 证书管理

```
SSL 证书获取策略 (按 Provider 分):

Vercel / Cloudflare:
  → 平台自动管理, 无需干预
  → 自定义域名也自动签发

Docker:
  → Caddy 自动 Let's Encrypt (推荐)
  → 或 Nginx + Certbot (定时续期 cron)

Kubernetes:
  → cert-manager + Let's Encrypt Issuer
  → HTTP-01 challenge (标准域名)
  → DNS-01 challenge (Wildcard 泛域名)
  → 自动续期 (到期前 30 天)
  → ClusterIssuer 配置:
      - name: letsencrypt-prod
      - email: admin@nova-studio.app
      - solvers: [HTTP-01, DNS-01 (Cloudflare)]

证书存储:
  → Kubernetes: K8s Secret (tls.crt + tls.key)
  → Docker: /etc/ssl/certs/<domain>/
  → Vercel/CF: 平台内部管理

证书续期:
  → Let's Encrypt 有效期 90 天
  → 自动在到期前 30 天续期
  → 续期失败 → 告警通知
```

## 13.6 回滚机制

### 回滚流程

```
用户点击 "Rollback"
      │
      ▼
┌─────────────────┐
│  1. 选择历史版本  │  展示最近 10 次成功部署
│  (Deploy History)│  每个版本显示: commit / 时间 / URL
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  2. 创建回滚记录  │  new Deployment { type: "rollback" }
│                  │  status: queued
│                  │  rollbackFrom: <currentDeploymentId>
│                  │  rollbackTo: <targetDeploymentId>
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  3. Provider     │  按 Provider 执行回滚:
│  Rollback        │
│                  │  Vercel:  POST /v13/deployments/{oldId}/promote
│                  │  CF:      POST .../deployments/{oldId}/rollback
│                  │  Docker:  docker stop && docker run <old-image>
│                  │  K8s:     kubectl rollout undo (滚动回滚)
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  4. 健康检查     │  验证回滚后的版本是否正常
│                  │  GET / → 200 OK × 3
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  5. 更新环境状态 │  environment.currentDeploymentId = oldId
│  记录历史        │  发布 rollback 事件
│                  │  通知用户
└─────────────────┘
```

### 回滚安全约束

| 约束 | 说明 |
|---|---|
| 只能回滚到成功部署 | status="live" 的部署才能作为回滚目标 |
| 回滚保留环境变量 | 回滚不改变环境变量配置, 只回滚代码 |
| 数据库不可回滚 | 回滚仅回滚应用代码, 数据库变更需手动处理 |
| 回滚也产生记录 | 回滚操作本身也是一条 deployment 记录 (type=rollback) |
| 回滚间隔限制 | 连续回滚需间隔 30s, 防止频繁操作 |
| 保留版本数 | 最近 20 次部署可回滚, 更早的自动清理 (K8s revisionHistoryLimit) |

## 13.11 Production Promotion (提升到生产)

### Promotion 流程

```
Preview/Staging 部署成功
       │
       ▼
┌──────────────────┐
│ 用户点击          │  "Promote to Production"
│ "Promote"        │  在 Preview 部署卡片上
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 确认对话框        │  "Are you sure you want to promote
│                  │   this deployment to production?"
│                  │  要求二次确认 (production 郲署保护)
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 创建 Promotion    │  new Deployment {
│ 记录              │    type: "promote",
│                  │    environmentId: production,
│                  │    artifactId: <preview.artifactId>,
│                  │    promotedFrom: <previewDeploymentId>

└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 复用构建产物      │  不重新构建, 直接使用 Preview 的 artifact
│ (Skip Build)     │  跳到 Deploy 阶段
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 部署到 Production │  按目标平台部署
│ (蓝绿部署)        │  K8s: 新版本 RollingUpdate
│                  │  Vercel: promote deployment
│                  │  Docker: 替换容器
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 健康检查 + 切换   │  验证新版本 → 切换流量
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 完成 + 通知       │  Production 环境更新
│                  │  通知团队
└──────────────────┘
```

### Promotion 优势

- **不重新构建**: 直接复用 Preview 的构建产物，确保一致性
- **快速上线**: 跳过 Build 阶段，30 秒内完成
- **版本一致**: Preview 验证通过 = Production 部署的代码

## 13.12 构建缓存策略

```
构建缓存层级:
  │
  ├── 1. 依赖缓存 (pnpm store)
  │   - 挂载 /root/.local/share/pnpm/store 为 Volume
  │   - 同项目同 lockfile 命中 → 秒级 install
  │   - 缓存 Key: hash(pnpm-lock.yaml)
  │
  ├── 2. Next.js 构建缓存 (.next/cache)
  │   - 挂载 /app/.next/cache 为 Volume
  │   - 增量构建, 只编译变更文件
  │   - 缓存 Key: hash(src/**) + hash(config)
  │
  ├── 3. Docker Layer 缓存
  │   - base image layer (Node.js + pnpm) → 永久缓存
  │   - dependency layer (pnpm install) → lockfile 不变则命中
  │   - build layer → 代码变更才重建
  │
  └── 4. 产物缓存
      - 构建产物存储在 S3/R2
      - 相同 commit hash → 直接复用, 跳过构建
      - Promotion 时复用 Preview 构建产物

缓存命中率统计:
  - 首次构建:    0% (全量构建)
  - 依赖未变:   ~90% (跳过 install)
  - 代码微调:   ~60% (增量编译)
  - 相同 commit: 100% (直接复用产物)
```

## 13.13 安全与隔离

### 构建沙箱隔离

```
Sandbox Pod 安全策略:
  │
  ├── 容器隔离
  │   - 使用 gVisor / Kata Containers (强隔离)
  │   - readOnlyRootFilesystem: true (只读根文件系统)
  │   - runAsNonRoot: true (非 root 运行)
  │   - allowPrivilegeEscalation: false
  │   - capabilities: drop ALL
  │
  ├── 网络隔离
  │   - NetworkPolicy: 仅允许出站到 npm registry + 部署平台 API
  │   - 禁止访问内部服务 (数据库 / Redis / 其他 Pod)
  │   - DNS 限制: 仅解析 npmjs.org / vercel.com / cloudflare.com 等
  │
  ├── 资源限制
  │   - CPU: 2 core (limit) / 0.5 core (request)
  │   - Memory: 2GB (limit) / 512MB (request)
  │   - Disk: 5GB (tmpfs)
  │   - 构建超时: 10 分钟 (超时自动 kill)
  │
  ├── 密钥注入
  │   - 环境变量通过 K8s Secret 注入 (临时, 构建后销毁)
  │   - npm token: 仅可读 (publish=false)
  │   - 构建日志脱敏: 自动过滤 secret 值
  │
  └── 清理
      - 构建完成后自动删除 Pod
      - 临时 Volume 自动回收
      - 构建产物上传后从本地删除
```

### 敏感信息保护

| 信息 | 存储方式 | 访问控制 |
|---|---|---|
| 平台 API Token | AES-256 加密存储 | 仅 Deploy Service 运行时解密 |
| 环境变量 (Secret) | K8s Secret / 加密存储 | 构建时注入, 日志自动脱敏 |
| SSL 私钥 | K8s Secret / 文件系统加密 | 仅 Ingress Controller 可读 |
| 构建日志 | PostgreSQL (自动过滤敏感值) | 仅项目成员可读 |
| Kubeconfig | AES-256 加密存储 | 仅 K8s Provider 运行时解密 |

## 13.17 错误处理与恢复

### 错误分类与处理

| 错误阶段 | 错误类型 | 处理策略 |
|---|---|---|
| Pre-deploy | 权限不足 | 拒绝部署, 返回 403 |
| Pre-deploy | 环境有进行中部署 | 拒绝部署, 返回 409 |
| Pre-deploy | 环境变量缺失 | 警告但继续, 使用默认值 |
| Build | pnpm install 失败 | 标记 failed, 保存日志 |
| Build | 编译错误 | 标记 failed, 高亮错误行 |
| Build | 构建超时 (>10min) | Kill Pod, 标记 failed (timeout) |
| Build | OOM (内存不足) | Kill Pod, 标记 failed (OOM) |
| Deploy | 平台 API 错误 | 重试 3 次, 间隔 5s |
| Deploy | 平台认证失败 | 标记 failed, 提示检查 Token |
| Domain | DNS 验证超时 | 标记 failed, 允许手动重试 |
| Domain | SSL 签发失败 | 标记 failed, 自动重试 Let's Encrypt |
| Health | 健康检查失败 | 自动回滚到上一版本 |
| Health | 超时 (>5min) | 标记 failed, 自动回滚 |

### 自动回滚触发条件

```
自动回滚触发:
  1. 健康检查连续 5 次失败 (间隔 10s)
  2. 部署后 5 分钟内未变为 live
  3. K8s: Rollout 状态为 Failed
  4. Vercel: 部署状态为 ERROR

自动回滚流程:
  → 创建 rollback 部署记录
  → 回滚到 previous_deployment_id
  → 健康检查
  → 通过 → 通知用户 "已自动回滚"
  → 失败 → 标记环境为 error, 紧急通知
```
